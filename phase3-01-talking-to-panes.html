Talking to Panes You Are Not In

Lesson 1 of 15 Intermediate 20 min

Phase 3 flips the tmux pattern you learned in Phase 1. Instead of switching to a pane to interact with it, you send commands to panes you’re not looking at. This is the send-keys command — the foundation of everything the cockpit automates.

Prerequisites:

  • Completed Phase 1 (tmux fundamentals: sessions, panes, windows, detach/reattach)
  • Completed Phase 2 (Claude Code skills: Courses 1–7 + Bridge 7½)
  • Run setup-tasks.sh to bootstrap ~/work/_tasks/ with seed data
  • tmux running with Ctrl+J as prefix

Concepts

The send-keys Command

$ tmux send-keys -t <target> "<command>" Enter

The -t flag is a target address. tmux addressing works like a postal system:

AddressMeaningExample
sessionTarget a session by name-t tasks
session:windowTarget a window by number or name-t tasks:1
session:window.paneTarget a specific pane-t tasks:0.1

Pane numbering starts at 0. In a two-pane horizontal split, pane 0 is the top (or left) and pane 1 is the bottom (or right).

Why This Matters for the Cockpit

The startup script you’ll build in Lesson 7 creates a multi-window tmux session and then sends commands to specific panes — launching Claude in one, starting watchers in others, all without you manually switching to each pane.

Walkthrough

1 Set up your task data

Run the setup script if you haven’t already:

$ bash courses/phase3/setup-tasks.sh

Verify it worked:

$ ls ~/work/_tasks/inbox/ $ ls ~/work/_tasks/pending/

You should see 5 files in inbox and 6 in pending.

2 Build a 3-window layout

Create a session, rename windows, and navigate:

$ tmux new -s tasks

Rename the window with Ctrl+J , → type hq. Create a second window (Ctrl+J c), rename to inbox. Create a third, rename to pending. Return to HQ: Ctrl+J 0.

You now have three windows: hq (0), inbox (1), pending (2).

3 Send commands to other windows

From the hq window, send a command to inbox without leaving:

$ tmux send-keys -t tasks:inbox "ls ~/work/_tasks/inbox/" Enter

Send another to pending:

$ tmux send-keys -t tasks:pending "ls -lt ~/work/_tasks/pending/" Enter

Switch to each window to verify: Ctrl+J 1, Ctrl+J 2, Ctrl+J 0.

4 Target panes within a window

Split the HQ window: Ctrl+J %. Now from the left pane, send to the right:

$ tmux send-keys -t tasks:hq.1 "echo 'Hello from pane 0!'" Enter

The echo runs in the right pane. Try sending a real command:

$ tmux send-keys -t tasks:hq.1 "wc -l ~/work/_tasks/pending/*.md" Enter
5 Chain commands

Send multiple commands in sequence:

$ tmux send-keys -t tasks:inbox "echo '--- Inbox Status ---'" Enter $ tmux send-keys -t tasks:inbox "echo 'Count:' && ls ~/work/_tasks/inbox/ | wc -l" Enter $ tmux send-keys -t tasks:inbox "echo 'Files:' && ls ~/work/_tasks/inbox/" Enter

Switch to the inbox window (Ctrl+J 1) to see all three commands executed in order.

Verification Checklist

  • ~/work/_tasks/ has 4 subdirectories with seed files (5 inbox, 6 pending, 2 archive, 2 log)
  • You have a tasks session with 3 named windows (hq, inbox, pending)
  • From hq, you can send-keys to the inbox window and see results there
  • From hq, you can send-keys to a specific pane (.0 or .1) within a split window
  • You can chain multiple send-keys calls to build up a command sequence in a remote pane

Design Rationale

Why send-keys instead of just running commands directly? Because the cockpit startup script (Lesson 7) needs to configure multiple panes from a single shell script. It can’t “be in” each pane — it creates them and then talks to them remotely.

Why named windows? Names like inbox and pending are stable addresses. Window numbers can shift if you create or destroy windows. tmux send-keys -t tasks:inbox "ls" Enter works regardless of window ordering.

Phase 1 Callback: This lesson builds on Lesson 3 (Split Panes), Lesson 6 (Windows), and Lesson 9 (Headless Sessions). You learned to create panes and windows — now you’re addressing them remotely.

What’s Next: Lesson 2: Capture Pane teaches the opposite of send-keys — reading output from panes you’re not in. Together, send and capture are the two primitives that make the cockpit work.

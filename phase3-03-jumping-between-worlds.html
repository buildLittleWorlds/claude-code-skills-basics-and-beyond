Jumping Between Worlds

Lesson 3 of 15 Intermediate 20 min

Real work spans multiple contexts: your task cockpit in one session, a coding project in another. tmux handles this with multiple simultaneous sessions, and this lesson teaches the rapid-switching workflow that the cockpit is designed to support.

Prerequisites:

  • Completed Lessons 1–2 (send-keys, capture-pane)
  • The tasks tmux session from previous lessons
  • Seed data in ~/work/_tasks/

Concepts

Multi-Session Architecture

Session: tasks ┌────────┬─────────────────┐ │ Claude │ Shell │ ← Task management └────────┴─────────────────┘ Session: coding ┌────────┬─────────────────┐ │ Editor │ Tests / Server │ ← Development work └────────┴─────────────────┘

Session Switching Commands

CommandWhat it does
Ctrl+J sInteractive session list — select with arrow keys
Ctrl+J (Switch to previous session
Ctrl+J )Switch to next session
tmux switch-client -t <name>Switch directly to a named session

Walkthrough

1 Create dual sessions

From within your tasks session, create a second detached session:

$ tmux new-session -d -s coding $ tmux list-sessions

You should see both coding and tasks.

2 Practice rapid switching

Use the session switcher: Ctrl+J s. Select coding and press Enter.

Practice the rapid switch cycle:

Ctrl+J ) # → tasks session (check what you need to work on) Ctrl+J ) # → coding session (back to work) Ctrl+J ( # → tasks session (switched back) Ctrl+J ( # → coding session (switched forward)

This is the switch-check-switch rhythm: jump to tasks, see what’s next, jump back to coding.

3 The manual “next task” query

In the tasks session, manually query your pending tasks:

$ for f in ~/work/_tasks/pending/*.md; do title=$(grep "^title:" "$f" | sed 's/title: //') due=$(grep "^due:" "$f" | sed 's/due: //') priority=$(grep "^priority:" "$f" | sed 's/priority: //') echo "$due $priority $title" done | sort

This works, barely. No error handling, fragile parsing, no color. The /next skill in Lesson 4 replaces this with a single command.

4 Cross-session commands

From tasks, send a command to coding:

$ tmux send-keys -t coding "echo 'Reminder: fix API timeout'" Enter

From coding, capture what’s in the tasks HQ:

$ tmux capture-pane -t tasks:hq.0 -p -S -10

Verification Checklist

  • You can create multiple tmux sessions and switch between them
  • Ctrl+J s opens the session switcher, Ctrl+J ( and ) cycle through sessions
  • You can send-keys and capture-pane across sessions
  • You ran the manual “next task” query and understand what /next will replace
  • You practiced the switch-check-switch rhythm between tasks and coding sessions

Design Rationale

Why separate sessions instead of windows? Sessions represent distinct contexts. When you switch sessions, you get a completely different set of windows and panes — a clean mental separation. The cockpit uses one session for task management and lets you run other sessions for actual work.

Why show the manual query? So you feel the pain that skills solve. Writing bash to parse YAML frontmatter is error-prone and unreadable. The /next skill in Lesson 4 replaces this with a single slash command.

Phase 1 Callback: This builds on Lesson 2 (Detach and Reattach) and Lesson 6 (Windows). You’re managing multiple live sessions without needing to detach — it’s a level up in the hierarchy: terminal > session > window > pane.

What’s Next: Lessons 1–3 gave you the manual building blocks. Starting in Lesson 4: /next Skill, you’ll replace the manual query with a proper Claude Code skill — the transition from bash one-liners to AI-powered workflows.

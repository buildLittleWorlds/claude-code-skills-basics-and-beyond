Side-Gigs Domain

Lesson 13 of 15 Advanced 20 min

No new artifacts — this is an integration test. Add freelance, content creation, and professional development tasks, then verify the entire pipeline handles them correctly.

Prerequisites:

  • Completed Lessons 4–12 (all skills, hooks, and cockpit layout working)
  • All 5 skills installed and tested with the original seed data

What’s Being Tested

The question: does the system you built generalize, or does it only work for the specific seed data you started with?

The side-gigs data adds freelance (client invoicing, project deadlines), content creation (blog posts, conference talks), and professional development (memberships, portfolio updates).

Walkthrough

1 Add the side-gigs seed data
$ cp courses/phase3/lesson-13-side-gigs/seed-tasks-sidegigs/inbox/*.md ~/work/_tasks/inbox/ $ cp courses/phase3/lesson-13-side-gigs/seed-tasks-sidegigs/pending/*.md ~/work/_tasks/pending/
2 Triage new inbox items
/triage

Claude processes the new inbox items. Check that routing rules handle freelance and personal project tasks correctly. If something is misclassified, update references/routing-rules.md.

3 Run the full pipeline

Test each skill with the expanded data set:

/next # Mixed data, sorted correctly? /done invoice-freelance Invoiced 12 hrs # Archive, log, hooks work? /overdue # Both domains in report? /today # Combined briefing accurate?
4 Assess and refine routing rules

If any classification was wrong, edit ~/.claude/skills/triage/references/routing-rules.md, move the misclassified item back to inbox, and re-triage. This iterative refinement is exactly how real skill development works.

Verification Checklist

  • Side-gigs seed data added (2 inbox + 3 pending)
  • /triage correctly classifies freelance and personal project tasks
  • /next shows all tasks sorted by due date
  • /done works on a side-gigs task (archive, log, popup)
  • /overdue includes both original and side-gigs overdue tasks
  • /today briefing reflects the expanded task set
  • Both hooks function correctly with the new data
  • Watchers and status bar reflect accurate counts

Design Rationale

Why a separate lesson for integration testing? Building features in isolation is different from running them together with real-world data variety. Unit tests verify components; integration tests verify the system.
